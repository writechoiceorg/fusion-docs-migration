The Query Pipelines API allows you to define parameters that should be applied to all queries processed by the system. A query pipeline consists of one or more stages. Query pipeline stages can be defined with the [Query Stages API](/managed-fusion/5.9/g5jma3/query-stages-api). See the [Query language cheat sheet](/managed-fusion/5.9/b4lyep/solr-query-language) for help constructing queries to send through a query pipeline.

For more information, [view the API specification.](?open-api=true)

## Query Pipeline Definition Properties

<table>
  <thead>
    <tr>
      <th>Property</th>
      <th>Description</th>
    </tr>
  </thead>
  <tbody>
    <tr>
      <td>
        id<br/>
        <em>Required</em>
      </td>
      <td>
        Required only when creating a new pipeline.
      </td>
    </tr>
    <tr>
      <td>
        stages<br/>
        <em>Required</em>
      </td>
      <td>
        <p>A JSON map of the stages to include with the pipeline. The stages can exist already or they can be defined while defining the pipeline.</p>
        <p>The stage must include the stage definitions:</p>
        <ul>
          <li><strong>id.</strong> the ID of the stage to include.</li>
          <li><strong>stageParams.</strong> Any parameters for the stage. These are only required for a stage that has been pre-defined and you wish to modify the properties.</li>
          <li><strong>type.</strong> The stage type. You can define a stage directly on a pipeline, or you can use a pre-existing stage. If you use a pre-existing stage, you must use “ref” as a reference to an existing stage.</li>
          <li><strong>skip.</strong> If the stage should be skipped during pipeline processing. By default, this is ‘false’, and does not need to be defined unless you would like to skip a stage in the future.</li>
        </ul>
        <p>If the stage supports setting other properties, as defined in the section <a href="/managed-fusion/5.9/ch321e/query-pipeline-stages">Query Pipeline Stages</a>, then those can be defined while defining the stage within the pipeline.</p>
      </td>
    </tr>
  </tbody>
</table>

## Examples

*Get the definition for the default query pipeline:*

<Tip>
You can omit the `default` pipeline name to list the definition for all pipelines, or include an app name in the URL (`/api/apps/APP_NAME/query-pipelines`) to list pipelines within a specific app.
</Tip>

**REQUEST**

```sh
curl -u USERNAME:PASSWORD https://EXAMPLE_COMPANY.b.lucidworks.cloud/api/query-pipelines/default
```

**RESPONSE**

```json
{
  "id" : "default",
  "stages" : [ {
    "type" : "recommendation",
    "id" : "recommendation",
    "numRecommendations" : 10,
    "useFq" : true,
    "numSignals" : 100,
    "aggrType" : "click@doc_id_s-query_s-filters_s",
    "skip" : false
  }, {
    "type" : "solr-query",
    "id" : "solr",
    "skip" : false
  } ]
}
```

*Create a new query pipeline, specifying a facet stage that is new and a solr-query stage that was previously defined:*

**REQUEST**

```sh
curl -u USERNAME:PASSWORD -X POST -H 'Content-Type: application/json' -d '{
    "id": "my-pipeline",
    "stages": [
        {
            "type": "facet",
            "id": "myFacets",
            "fieldFacets": [
                {
                    "field": "data_source_s"
                },
                {
                    "field": "author_s",
                    "minCount": 5,
                    "missing": true
                },
                {
                    "field": "isRetweet_b"
                }
            ],
            "skip": false
        },
        {
            "id": "mySolr",
            "httpMethod": "POST",
            "allowFederatedSearch": false,
            "preferredReplicaType": "pull",
            "type": "solr-query",
            "skip": false,
            "responseSignalsEnabled": true
        }
    }
}' https://EXAMPLE_COMPANY.b.lucidworks.cloud/api/apps/APP_NAME/my-pipeline/query-pipelines
```
