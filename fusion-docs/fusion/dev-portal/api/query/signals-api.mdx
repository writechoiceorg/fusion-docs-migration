The Signals API accepts a set of [signals](/fusion/5.9/466/signals-types-and-structures), encoded as JSON objects, for indexing into a signals collection.

For more information, [view the API specification.](?open-api=true)

Normally, signals are indexed just like ordinary documents, through a configured datasource and index pipeline. This API is provided for cases where it is more convenient to index signals directly.

To aggregate signals, use a [SQL aggregation](/fusion/5.9/8804/sql-aggregation-jobs) job, which is a kind of [Spark job](/fusion/5.9/187/spark-jobs).

<Tip>
You can manage aggregation jobs in the [Jobs manager](/fusion/5.9/184/jobs-and-scheduling#jobs-manager) and [Scheduler](/fusion/5.9/184/jobs-and-scheduling#scheduler) in the Admin UI, or with the [Spark jobs API](/fusion/5.9/312/spark-jobs-api). We recommend using the Admin UI.
</Tip>

See [Signal types and structure](/fusion/5.9/469/signals#signals-types-and-structure) to learn how to send well-formed signals to this API.

<Note>
This example uses the `commit=true` setting, which commits after every signal. The setting is not recommended in production and performance testing because it negatively impacts performance. Instead of sending the `commit` parameter, consider using no commit parameter and let the defaults of the Solr collection dictate the commit frequency.
</Note>

```bash
curl \
 -u USERNAME:PASSWORD -X POST -H 'Content-type:application/json' -d @- \
https://FUSION_HOST:FUSION_PORT/api/apps/APP_NAME/signals/COLLECTION_NAME?async=false&commit=true \
<<EOF
[
 {"params": {
      "query": "Televisiones Panasonic  50 pulgadas",
      "filterQueries": ["cat00000","abcat0100000", "abcat0101000", "abcat0101001"],
      "docId": "2125233" },
 "type":"click",
 "timestamp": "2011-09-01T23:44:52.53Z"
 },
 {"params": {
       "query": "Sharp",
       "filterQueries": ["cat00000", "abcat0100000", "abcat0101000", "abcat0101001"],
       "docId": "2009324" },
  "type":"click",
  "timestamp": "2011-09-05T12:25:37.42Z"
 }
]
EOF
```
